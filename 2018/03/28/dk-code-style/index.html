DK的通用代码规范

2018.03.28

通用编码规范针对所有可能使用到的编程语言.

总则

所有模糊的规定和认定都以此为标尺, 尽可能地契合此处阐述的规则.

  1. 编写简单的代码而不是复杂的代码.
  2. 优先考虑程序的可读性.
  3. 编写简短的代码而不是冗长的代码.
  4. 注释仅用于解释程序的编写思路,技巧和坑. 代码的功能, 参数的传递方式等, 尽可能用自身的名称描述, 而不是用注释.
  5. 尽可能封装高性能代码, 不要让别人修改经过高度优化的代码, 尤其是在大量优化使得可读性和可维护性严重下降时.

命名规范

类名, 结构体名, 函数名使用大驼峰.

变量名, getter, setter 使用小驼峰.

变量名和函数名中尽可能少地出现数字. 类名禁止出现数字, 除非该类仅用于存储数据.

其它(如命名空间)参考语言规范.

格式规范

所有双目运算符(包括赋值和初始化的等号)左右各一个空格. 尖括号和圆括号两边不能有空格, 除非与圆括号相邻的符号要求加空格.

对于lambda表达式所需的符号, 括号内侧不应有空格, lambda运算符两边各一个空格.

所有二元位运算均需使用圆括号, 不论此括号是否改变运算顺序. 一元运算符不需要.

表示程序执行的大括号有且仅有两种写法. 第一种是和后面的代码单独成一语句. 此时花括号内部必须有且仅有一个和括号相邻的空格.

第二种是大括号分别占据两行, 行内不能有其它内容.

表示数据初始化的大括号, 左花括号不换行, 此时左花括号前有一个空格.
右花括号的位置在最后一个数据之后, 可以视情况在前面插入一个空格.

因单独一行过长而导致的截断, 采用4个空格缩进. lambda表达式可以在lambda符号之后破行, 此时该lambda表达式不缩进.

过长的参数表和初始化列表应完全破行, 右圆括号接在最后一个参数之后, 除非数据的分组有着直观的意义:

对于没有大括号的语言以及上述语法会造成语义模糊或语法错误的语言, 大括号格式以语言推荐的方式为准.
